## Problem Statement

`pip_notification_service.py` calls `django.core.mail.send_mail` after an admin finalises a PIP. In production, `EMAIL_BACKEND` was unset — Django would have fallen back to SMTP with no host configured, silently dropping every email. `boto3` was listed in `requirements.txt` for SES but never wired to Django's email dispatch.

---

## Design Decisions

### Why `django-ses` over raw boto3?

| Criterion | `django-ses` | Raw boto3 in `EmailService` |
|-----------|-------------|---------------------------|
| Works with Django `send_mail` / `send_mass_mail` | ✅ | ❌ (bypass Django mail) |
| Works with Django `mail_admins` / `mail_managers` | ✅ | ❌ |
| Works with `django.contrib.auth` password reset | ✅ | ❌ |
| SES bounce/complaint webhook handler | ✅ built-in | ❌ manual |
| Test backend swap (locmem) | ✅ automatic | ❌ requires monkeypatching |
| Maintenance burden | Low (maintained Django package) | High |

`accounts/services/email_service.py` (auth welcome/password-reset) still uses raw boto3 — it pre-dates this feature. It will be migrated in a follow-up ticket to use `send_mail` so the same backend logic covers it.

### Why NOT SMTP relay?

SES SMTP interface exists but adds one network hop and requires storing SMTP credentials. Native SES API via boto3 is simpler, cheaper, and more observable.

## Remaining Work (S6–S8)

### S6: AWS SES Provisioning

Steps (one-time, done in AWS Console or via CloudFormation/Terraform):

1. **Verify sending domain** `featurefactory.io`:
   - Add SES DNS TXT record for domain verification.
   - Add DKIM CNAME records (3 records generated by SES).
   - Confirm in SES Console: `featurefactory.io` → Verified.

2. **Request SES production access** (exit sandbox):
   - AWS Support → Service Limit Increase → SES Sending Limits.
   - Justification: transactional, user-initiated, low volume.
   - Without this, emails only reach verified addresses.

3. **Create IAM policy `MimirSESSendEmail`**:
   ```json
   {
     "Version": "2012-10-17",
     "Statement": [{
       "Effect": "Allow",
       "Action": ["ses:SendEmail", "ses:SendRawEmail"],
       "Resource": "arn:aws:ses:us-east-1:<ACCOUNT>:identity/featurefactory.io"
     }]
   }
   ```

4. **Attach policy to EB instance role** `aws-elasticbeanstalk-ec2-role` (preferred over long-lived access keys).

5. **(Optional) Create SES Configuration Set** `mimir-transactional`:
   - CloudWatch event destination for delivery, bounce, complaint metrics.
   - Set `AWS_SES_CONFIGURATION_SET=mimir-transactional` in EB env vars.

### S7: EB Environment Variables

Set in AWS Console → Elastic Beanstalk → Environment → Configuration → Software:

| Variable | Value |
|----------|-------|
| `AWS_SES_REGION_NAME` | `us-east-1` |
| `DEFAULT_FROM_EMAIL` | `noreply@featurefactory.io` |
| `FRONTEND_URL` | `https://mimir.featurefactory.io` |
| `AWS_SES_CONFIGURATION_SET` | `mimir-transactional` (optional) |

Do **not** set `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` when using instance role.

### S8: Production Verification

1. Deploy new build.
2. Create a test PIP as admin user.
3. Admin finalise it → check Django logs for `Sending PIP decision email`.
4. Verify email arrives at submitter inbox.
5. Check SES Console → Send Statistics for bounce/complaint rates.
